# Actividad: definición del proyecto final

Antes de construir, tienen que elegir bien el problema.

En esta actividad van a proponer el proyecto que quieren desarrollar para el examen de XTECC21. La idea es conversar el alcance, ajustar expectativas y autorizar el proyecto antes de que empiecen a ejecutarlo.

Yo revisaré cada propuesta con ustedes. Podemos achicar, enfocar o cambiar el proyecto si hace falta. La meta es que cada equipo salga con un desafío claro, posible y útil.

## Objetivo

Definir una automatización para estudiantes que pueda convertirse en una app o herramienta funcional con ayuda de Codex.

## Trabajo en equipo

Conversen y elijan una situación real. No partan desde la tecnología. Partan desde una molestia, pérdida de tiempo, confusión o tarea repetitiva que ustedes conozcan.

Busquen un problema que puedan explicar en una frase:

> Estudiantes necesitan ______ porque hoy ______.

Después piensen qué parte de ese problema puede automatizarse de manera responsable.

## Entrega de la actividad

Cada equipo debe preparar una propuesta breve con:

- Nombre tentativo del proyecto.
- Integrantes del equipo.
- Problema que quieren resolver.
- Usuario principal.
- Situación concreta en que aparece el problema.
- Qué tarea quieren automatizar.
- Qué forma tendría la solución: app web, mobile, bot, extensión, flujo con Codex u otra herramienta.
- Qué tendría que hacer el prototipo mínimo para considerarlo logrado.
- Qué datos, cuentas o permisos necesitarían.
- Qué riesgos ven: privacidad, dificultad técnica, dependencia de plataformas, tiempo o alcance.

La propuesta debe ser corta. Una página basta si está bien pensada.

## Preguntas para pensar

Usen estas preguntas para aterrizar la idea:

- ¿A quién le duele este problema?
- ¿Cuándo ocurre?
- ¿Cómo se resuelve hoy?
- ¿Qué parte se repite siempre?
- ¿Qué decisión o acción podría ayudar a tomar Codex?
- ¿Qué sería una primera versión útil?
- ¿Qué cosa no vamos a intentar resolver todavía?

La última pregunta es clave. Un buen alcance también dice qué queda fuera.

## Revisión y autorización

Después de revisar la propuesta, conversaremos cada proyecto.

La revisión puede terminar en una de estas decisiones:

- Aprobado: el equipo puede empezar a construir.
- Aprobado con ajustes: el equipo debe cambiar alcance, foco o formato antes de construir.
- Replantear: la idea todavía es muy amplia, poco clara o difícil de ejecutar en el tiempo disponible.

No empiecen a construir la versión final antes de tener el proyecto autorizado. Sí pueden explorar ejemplos, referencias, herramientas y datos mientras afinamos la idea.

## Criterios para autorizar un proyecto

Voy a mirar si:

- El problema es real y entendible.
- El usuario está claro.
- La solución tiene un alcance razonable para el curso.
- Codex puede aportar de forma concreta.
- El equipo puede mostrar una demo funcional al final.
- La propuesta cuida privacidad y datos personales.

La mejor propuesta no es la más grande. Es la que permite construir algo que se pueda probar, explicar y mejorar.
